# Easy UI5 gets a tune-up: new release flow, dedicated config home, prettier banner — and a few things you'll want to migrate

_Draft for the SAP Community blog — June 26, 2026_

If you've been scaffolding UI5 apps with `yo easy-ui5` for a while, you'll know that the generator has quietly been doing a lot of plumbing on your behalf: discovering plugin generators on GitHub, running `npm install`, calling out to `git`, asking you a handful of questions and rendering templates. The last two releases — **3.10.0** and the just-published **3.10.1** patch — modernize the bits underneath that surface without changing the day-to-day flow. This post walks through what's new, what got better, and the few mandatory adoptions you should plan for.

> **TL;DR** — Easy UI5 now uses Changesets for releases, has its own `~/.easyui5rc.json` config home (so you can finally stop sneaking keys into `~/.npmrc`), sports a branded ASCII banner with a tagged logger, and ships fixes for two interactive-flow regressions. The minimum Node.js version is **20.19**. If you have `easy-ui5_*` keys in `~/.npmrc`, you have one release window to move them.

## What's new in 3.10.0

### A real release pipeline — Changesets and OIDC publishing

The legacy tag-based publish workflow is gone. Every PR now records its intended semver bump and a short summary in a `.changeset/*.md` file, and merging to `main` automatically opens a **Version Packages** PR. Merging _that_ PR publishes to npm via **OIDC Trusted Publishing** — no long-lived npm token sitting in repository secrets.

For contributors this means one extra step on PRs:

```bash
npm run changeset
```

The script asks what bump (`patch` / `minor` / `major`) and what summary to write. CI fails PRs that don't include a changeset (with a deliberate exception for chore-only or docs-only changes via the empty-changeset script).

The previous `release:changelog`, `postversion` hook and the `conventional-changelog-cli` devDependency are all gone — `CHANGELOG.md` is now generated by `@changesets/changelog-github` and contains live PR and commit links.

### A dedicated config home: `~/.easyui5rc.json`

For years, `generator-easy-ui5` read its configuration out of `~/.npmrc` via keys like `easy-ui5_ghAuthToken=…`. That worked, but it polluted npm's own config namespace — and npm 11 already warns about unknown keys; npm 12 will reject them outright.

3.10.0 introduces a proper config home:

- **`~/.easyui5rc.json`** (preferred) or `~/.easyui5rc` (INI) for the generator's options:
  - `ghAuthToken`, `ghBaseUrl`, `pluginsHome`, `addGhBaseUrl`, `addGhOrg`, `addSubGeneratorPrefix`
- The lookup walks from `cwd` upwards to your home directory — an **ESLint-style cascade**. Any file along the way can set `"root": true` to stop the walk.
- New `EASY_UI5_*` environment variables for CI, e.g. `EASY_UI5_GH_AUTH_TOKEN`, override every config file.

Internally we also stopped shelling out to `npm config list --json` to read these values, removing a child-process dependency and shaving ~100–300 ms off generator startup.

### A branded banner and a tagged logger

The standard Yeoman `yosay` greeting has been replaced with a branded **EASY UI5** ASCII banner — the wordmark is painted in a six-tone rainbow with a small magenta heart anchored top-right, and the version number sits on the last wordmark row. It's a tiny thing, but it's nice to look at, and dropping `yosay` from the runtime dependencies trimmed one more transitive tree.

More substantively, every diagnostic line the generator emits now carries a fixed-width severity tag:

```
[INFO ]  Installing dependencies in ./my-app
[OK   ]  Generated 14 files
[WARN ]  easy-ui5_ghAuthToken in ~/.npmrc is deprecated — move it to ~/.easyui5rc.json
```

- `INFO` / `WARN` / `ERROR` / `OK` / `DEBUG` — each colour-coded
- `WARN` and `ERROR` go to **stderr** so they survive `stdout` redirects
- `DEBUG` is hidden unless you pass `--verbose`

If you script around the generator's output, the leading `[INFO ]`-style prefixes are stable and easy to grep for.

### Test matrix covers yeoman-generator v4 through v8

The integration-test suite now runs against **every supported `yeoman-generator` major** (4, 5, 6, 7, 8). Each fixture lives in `test/generator-ui5-test-v<N>/` and produces a distinctively-named template file, so a regression in any major surfaces immediately. Adding a future major is a one-line change to the data-driven driver.

This already paid off: it caught a latent `chalk`-not-imported bug in the v4/v5 fixtures and a real production regression on v8 (see below).

### Toolchain bumps

- **Minimum Node.js is 20.19** (was `>=20`), matching `eslint@10`'s engines field.
- `yeoman-generator` → `^8.2.2`, `eslint` → `^10.5.0`.
- New `eslint-plugin-n` and `eslint-plugin-security` plus a strict CI variant (`npm run lint:ci`, `--max-warnings 0`) for supply-chain hygiene.

### A subtle but nasty fix: `spawnCommand` → `spawn`

If you tried to use a sub-generator on `yeoman-generator` 8 and got a cryptic `Command failed with exit code 1: npm`, this one's for you. In v8 the spawn API was reshuffled: `spawnCommand(name, opts)` now takes a shell _string_ (no args array), while `spawn(name, args, opts)` is the args-array variant. The generator was still calling `spawnCommand` with an args array — execa silently dropped the args and invoked `npm` with nothing, which prints help and exits 1.

All four call sites (`_npmInstall` and the three `git` calls in `end()`) now use `spawn` / `spawnSync`, and there's a regression-guard test that fails if the legacy API reappears anywhere under `generators/`.

## What's new in 3.10.1 (today's patch)

Two small but visible interactive-flow fixes:

1. **Dropped `--prefer-offline` from the internal `npm install`.** The flag could prevent the install from completing properly when the offline cache was incomplete or stale — leaving users with a half-scaffolded project and a confusing error.
2. **Replaced the deprecated inquirer `list` prompt with `select`.** Inquirer printed a deprecation warning every time you ran `yo easy-ui5` and got to the generator-picker question. That noise is gone.

## Mandatory adoptions — please read

These are the items that _will_ affect you if you don't act on them.

### 1. Move `easy-ui5_*` keys out of `~/.npmrc`

If your `~/.npmrc` contains lines like:

```ini
//easy-ui5_ghAuthToken=ghp_…
easy-ui5_ghBaseUrl=https://github.example.corp/api/v3
```

…move them into a new `~/.easyui5rc.json`:

```json
{
  "ghAuthToken": "ghp_…",
  "ghBaseUrl": "https://github.example.corp/api/v3",
  "pluginsHome": "ui5-community"
}
```

3.10.x **still reads the old keys** but prints a one-time stderr deprecation notice. **The next release will drop the compatibility shim entirely**, and npm 12 will reject the unknown keys anyway. Migrate now while it's a 30-second copy-paste.

### 2. Make sure your Node.js is `>=20.19`

The previous floor was `>=20.0.0`. If you're on a stale 20.x patch line, `npm install -g generator-easy-ui5` will refuse to install. `nvm install --lts` or `nvm install 20.19` is the quickest fix.

### 3. Maintainers publishing locally: run `npm run embed` first

This only affects people releasing the package themselves (the CI workflow is the intended path and handles this automatically). The `--embed` plugin-generator snapshotting step has been pulled out of the `prepublishOnly` hook and into an explicit workflow step that runs **before** Changesets publishes, with a properly authenticated GitHub fetch. If you publish locally, you now need:

```bash
npm run embed
npm run release-publish
```

See the **Release Life-Cycle** section in `CONTRIBUTING.md` for the full story.

### 4. PR workflow: every functional PR needs a changeset

Run `npm run changeset` before opening (or before pushing the final commit on) any PR that should bump the published version. CI will block merges that lack a changeset entry, with a deliberate escape hatch for docs-only and chore-only changes (`npm run changeset:empty`).

## How to upgrade

Globally:

```bash
npm install -g generator-easy-ui5@latest
```

…or use `npx`:

```bash
npx yo@7.0.1 easy-ui5
```

If you previously ran the generator from a folder that has its own `~/.easyui5rc.json`, the cascade picks it up automatically — no further action.
